What Changed: The Async-First Pivot

The origin design had 2–3 synchronous REST round-trips per stage boundary adding up to ~1.4–2.8 seconds of tracking overhead per pipeline run. This final version replaces that with an OCI Queue async buffer between business integrations and the ATP persistence layer, reducing tracking overhead to ~20–40ms per enqueue call — a 75–85% latency reduction. Two additional architectural shifts enable full async:

Shift	Previous	Final
correlationId source	Generated by PIPELINE_TRACKER_INIT (required sync call)	Generated locally in business integration via JS UUID — no sync dependency
Tracker invocation	Synchronous REST calls to 3 tracker integrations	Fire-and-forget enqueue to OCI Queue
ATP write timing	Immediate, inline with business flow	Decoupled, async via Queue Dispatcher
Failure of tracker	Could stall business flow during retries	Fully invisible to business flow — queue absorbs it

1. Design Principles (Async-First)

Principle	Implementation
Async-first	Business flow enqueues tracking events and moves on immediately — never waits for ATP writes
Local correlation	correlationId generated in business integration via JS UUID — zero sync dependency at entry
Full decoupling	OCI Queue is the hard boundary between business logic and persistence — ATP issues are invisible to business flows
Eventual consistency accepted	Tracking records reflect state within seconds of actual execution — acceptable for observability
Batch efficiency	Queue dispatcher processes messages in micro-batches — fewer DB round-trips than one-per-event sync
Resilience layered	Circuit breaker → DLQ → OCI Logging fallback — three safety nets before any data is lost
Developer minimal-friction	Enqueue calls are fire-and-forget with no response parsing needed — simpler than sync REST invoke pattern

3. OCI Queue Design
3.1 Queue Topology
Two dedicated queues separate high-volume log writes from structured tracking state updates:

[GAP-01] Both queues are configured with timeToLive = 86400 (24 hours). Messages older than
24 hours are stale — their pipeline has long since completed or failed. Without TTL, a
dispatcher outage followed by recovery would process hours-old events against already-closed
pipeline records, corrupting state machine data.

[ACC-04] Two distinct dead-letter stores exist — they are NOT the same thing and serve
different failure modes. See Section 5.1 Phase 4 for the complete two-store DLQ pattern:
  (A) OCI Queue native DLQ (oic-tracking-dlq): receives messages when maxDeliveryCount
      is exceeded — i.e., messages the dispatcher NEVER successfully dequeued.
  (B) OIC_TRACKING_DLQ ATP table (SP_WRITE_DLQ): application-level registry for messages
      the dispatcher DID dequeue but could not persist after all retries.

Code
Queue 1: oic-tracking-events
  ├── Messages:         PIPELINE_INIT, STAGE_UPDATE, PIPELINE_COMPLETE
  ├── Volume:           Lower (~4–10 msgs per pipeline run)
  ├── Priority:         State machine integrity — FIFO, ordered individual processing
  ├── timeToLive:       86400 seconds (24 hours)       [GAP-01]
  ├── maxDeliveryCount: 5  (→ oic-tracking-dlq after 5 failed delivery attempts)
  └── DLQ:             oic-tracking-dlq               (OCI Queue native DLQ)

Queue 2: oic-activity-logs
  ├── Messages:         ACTIVITY_LOG
  ├── Volume:           Higher (~10–50 msgs per pipeline run)
  ├── Priority:         Throughput — batch processing
  ├── timeToLive:       86400 seconds (24 hours)       [GAP-01]
  ├── maxDeliveryCount: 3  (lower tolerance — log writes are idempotent on retry)
  └── DLQ:             oic-tracking-dlq               (shared OCI Queue native DLQ)

-- [GAP-02] PARTITIONING STRATEGY — PRODUCTION RECOMMENDATION
-- ============================================================
-- For deployments processing > 10,000 pipeline runs/month, partition
-- OIC_PIPELINE_RUNS, OIC_STAGE_TRACKING, and OIC_ACTIVITY_LOG on
-- INITIATED_AT / LOG_TIMESTAMP using INTERVAL range partitioning.
--
-- BENEFIT: The nightly retention job (Section 9) becomes a single
-- DROP PARTITION — instantaneous, zero undo, zero lock contention —
-- instead of correlated DELETE subqueries scanning millions of rows.
--
-- ATP supports INTERVAL partitioning. Add to table DDL at creation:
--
--   PARTITION BY RANGE (INITIATED_AT)
--   INTERVAL (NUMTOYMINTERVAL(1,'MONTH'))
--   (PARTITION p_initial VALUES LESS THAN (TIMESTAMP '2026-01-01 00:00:00'));
--
-- Retention job then uses:
--   EXECUTE IMMEDIATE 'ALTER TABLE OIC_PIPELINE_RUNS DROP PARTITION ' || v_partition_name;
-- instead of DELETE. Apply same pattern to OIC_STAGE_TRACKING (on CREATED_AT)
-- and OIC_ACTIVITY_LOG (on LOG_TIMESTAMP).
--
-- NOTE: Cannot add interval partitioning to an existing non-partitioned table
-- via ALTER TABLE. Plan partitioning at initial schema creation.
-- ============================================================

CREATE OR REPLACE PACKAGE BODY OIC_TRACKER_PKG AS

-- ------------------------------------------------------------
-- SP_INIT_PIPELINE
-- Called async by dispatcher on PIPELINE_INIT message
-- [R-02] p_stage_manifest DATA FLOW CLARIFICATION:
--   This parameter is NOT sourced from the PIPELINE_INIT queue message.
--   'stageManifest' was removed from the queue payload in ACC-05 to eliminate
--   an inconsistency vector (stale stage definitions in in-flight messages).
--   Instead, the dispatcher reads the authoritative manifest as a CLOB from
--   OIC_INTEGRATION_CONFIG.STAGE_MANIFEST_JSON keyed by p_integration_id,
--   then passes it to this procedure directly. The queue message payload
--   carries only: businessKey, initiatedBy, oicInstanceId, environment.
--   OIC_INTEGRATION_CONFIG is the single source of truth for pipeline topology.
-- ------------------------------------------------------------
PROCEDURE SP_INIT_PIPELINE (
    p_correlation_id   IN  VARCHAR2,
    p_integration_id   IN  VARCHAR2,
    p_business_key     IN  VARCHAR2,
    p_initiated_by     IN  VARCHAR2,
    p_initiated_at     IN  VARCHAR2,
    p_oic_instance_id  IN  VARCHAR2,
    p_environment      IN  VARCHAR2,  -- [R-04] PROD | UAT | DEV from queue payload
    p_stage_manifest   IN  CLOB,
    p_status           OUT VARCHAR2,
    p_error_msg        OUT VARCHAR2
) AS
    v_flow_name       VARCHAR2(500);
    v_total_stages    NUMBER := 0;
    v_init_ts         TIMESTAMP;
BEGIN
    -- Parse ISO8601 timestamp from queue message (preserve original enqueue time)
    v_init_ts := TO_TIMESTAMP(p_initiated_at, 'YYYY-MM-DD"T"HH24:MI:SS.FF3"Z"');

    SELECT FLOW_DISPLAY_NAME INTO v_flow_name
    FROM   OIC_INTEGRATION_CONFIG
    WHERE  INTEGRATION_ID = p_integration_id;

    -- [BUG-06 FIX] MERGE instead of INSERT — idempotent against OCI Queue at-least-once redelivery.
    -- If dispatcher crashes after SP call but before queue DELETE, redelivery is safe.
    MERGE INTO OIC_PIPELINE_RUNS pr
    USING (SELECT p_correlation_id AS cid FROM DUAL) src
    ON (pr.CORRELATION_ID = src.cid)
    WHEN NOT MATCHED THEN
        INSERT (CORRELATION_ID, INTEGRATION_ID, FLOW_DISPLAY_NAME,
                ENVIRONMENT, BUSINESS_KEY, INITIATED_BY, INITIATED_AT,
                OIC_INSTANCE_ID, OVERALL_STATUS, CREATED_AT)
        VALUES (p_correlation_id, p_integration_id, v_flow_name,
                NVL(p_environment, 'PROD'),  -- [R-04] defaults to PROD if absent
                p_business_key, p_initiated_by, v_init_ts,
                p_oic_instance_id, 'IN_PROGRESS', SYSTIMESTAMP);

    -- [BUG-02 FIX] isMandatory is a JSON boolean (true/false).
    -- JSON_TABLE extracts it as VARCHAR2(10) then CASE maps it to Y/N.
    -- VARCHAR2(1) extraction would truncate 'true' to 't' — incorrect.
    -- [BUG-06 FIX] WHERE NOT EXISTS guard prevents duplicate stage rows
    -- if PIPELINE_INIT message is redelivered by OCI Queue.
    INSERT INTO OIC_STAGE_TRACKING (
        TRACKING_ID, CORRELATION_ID, STAGE_ID, STAGE_NAME,
        STAGE_ORDER, IS_MANDATORY, STATUS, CREATED_AT
    )
    SELECT
        SYS_GUID(),
        p_correlation_id,
        jt.stage_id,
        jt.stage_name,
        jt.stage_order,
        CASE WHEN jt.is_mandatory_raw IN ('true','1') THEN 'Y' ELSE 'N' END,
        'PENDING',
        SYSTIMESTAMP
    FROM JSON_TABLE(p_stage_manifest, '$.stages[*]'
        COLUMNS (
            stage_id         VARCHAR2(50)  PATH '$.stageId',
            stage_name       VARCHAR2(200) PATH '$.stageName',
            stage_order      NUMBER        PATH '$.stageOrder',
            is_mandatory_raw VARCHAR2(10)  PATH '$.isMandatory'  -- boolean → string, mapped below
        )
    ) jt
    WHERE NOT EXISTS (
        SELECT 1 FROM OIC_STAGE_TRACKING
        WHERE  CORRELATION_ID = p_correlation_id
        AND    STAGE_ID        = jt.stage_id
        AND    ATTEMPT_NUMBER  = 1
    );

    SELECT COUNT(*) INTO v_total_stages
    FROM   OIC_STAGE_TRACKING
    WHERE  CORRELATION_ID = p_correlation_id;

    UPDATE OIC_PIPELINE_RUNS
    SET    TOTAL_STAGES = v_total_stages
    WHERE  CORRELATION_ID = p_correlation_id;

    COMMIT;
    p_status    := 'SUCCESS';
    p_error_msg := NULL;

EXCEPTION WHEN OTHERS THEN
    ROLLBACK;
    p_status    := 'ERROR';
    p_error_msg := SQLERRM;
END SP_INIT_PIPELINE;


-- ------------------------------------------------------------
-- SP_UPDATE_STAGE
-- State machine enforced at DB layer
-- ------------------------------------------------------------
PROCEDURE SP_UPDATE_STAGE (
    p_correlation_id   IN  VARCHAR2,
    p_stage_id         IN  VARCHAR2,
    p_new_status       IN  VARCHAR2,
    p_error_code       IN  VARCHAR2,
    p_error_message    IN  VARCHAR2,
    p_payload_ref      IN  VARCHAR2,
    p_metadata         IN  CLOB,
    p_status           OUT VARCHAR2,
    p_error_msg        OUT VARCHAR2
) AS
    v_current_status  VARCHAR2(20);
    v_tracking_id     VARCHAR2(36);
    v_attempt         NUMBER;
    v_valid           NUMBER := 0;
BEGIN
    SELECT STATUS, TRACKING_ID, ATTEMPT_NUMBER
    INTO   v_current_status, v_tracking_id, v_attempt
    FROM   OIC_STAGE_TRACKING
    WHERE  CORRELATION_ID  = p_correlation_id
    AND    STAGE_ID         = p_stage_id
    AND    ATTEMPT_NUMBER   = (
               SELECT MAX(ATTEMPT_NUMBER)
               FROM   OIC_STAGE_TRACKING
               WHERE  CORRELATION_ID = p_correlation_id
               AND    STAGE_ID        = p_stage_id
           );

    -- State machine validation matrix
    SELECT COUNT(*) INTO v_valid FROM DUAL WHERE
        (v_current_status = 'PENDING'     AND p_new_status = 'IN_PROGRESS') OR
        (v_current_status = 'IN_PROGRESS' AND p_new_status = 'COMPLETED')   OR
        (v_current_status = 'IN_PROGRESS' AND p_new_status = 'FAILED')      OR
        (v_current_status = 'IN_PROGRESS' AND p_new_status = 'SKIPPED')     OR
        (v_current_status = 'FAILED'      AND p_new_status = 'IN_PROGRESS') OR
        (v_current_status = 'PENDING'     AND p_new_status = 'SKIPPED');

    IF v_valid = 0 THEN
        p_status    := 'INVALID_TRANSITION';
        p_error_msg := 'Transition ' || v_current_status
                       || ' → ' || p_new_status || ' not permitted';
        RETURN;
    END IF;

    -- On retry: insert new attempt row
    -- [BUG-03 FIX] STARTED_AT is populated on the new attempt row so that
    -- DURATION_MS is computable when this attempt reaches COMPLETED/FAILED/SKIPPED.
    -- Previously missing STARTED_AT caused NULL DURATION_MS on all retried stages.
    IF v_current_status = 'FAILED' AND p_new_status = 'IN_PROGRESS' THEN
        INSERT INTO OIC_STAGE_TRACKING (
            TRACKING_ID, CORRELATION_ID, STAGE_ID, STAGE_NAME,
            STAGE_ORDER, IS_MANDATORY, STATUS, ATTEMPT_NUMBER, STARTED_AT, CREATED_AT
        )
        SELECT SYS_GUID(), p_correlation_id, STAGE_ID, STAGE_NAME,
               STAGE_ORDER, IS_MANDATORY, 'IN_PROGRESS', v_attempt + 1,
               SYSTIMESTAMP, SYSTIMESTAMP  -- STARTED_AT = now (attempt begins immediately)
        FROM   OIC_STAGE_TRACKING
        WHERE  TRACKING_ID = v_tracking_id;
    ELSE
        UPDATE OIC_STAGE_TRACKING
        SET    STATUS               = p_new_status,
               STARTED_AT          = CASE WHEN p_new_status = 'IN_PROGRESS'
                                          THEN SYSTIMESTAMP ELSE STARTED_AT END,
               COMPLETED_AT        = CASE WHEN p_new_status IN
                                          ('COMPLETED','FAILED','SKIPPED')
                                          THEN SYSTIMESTAMP ELSE NULL END,
               -- [BUG-01 FIX] EXTRACT(SECOND FROM interval) returns only the 0-59 seconds
               -- component — not total seconds. A 2m30s stage would give 30000ms, not 150000ms.
               -- Full multi-component extraction is required for correctness.
               DURATION_MS         = CASE WHEN p_new_status IN
                                          ('COMPLETED','FAILED','SKIPPED')
                                          THEN ROUND(
                                               EXTRACT(DAY    FROM (SYSTIMESTAMP - STARTED_AT)) * 86400000 +
                                               EXTRACT(HOUR   FROM (SYSTIMESTAMP - STARTED_AT)) * 3600000  +
                                               EXTRACT(MINUTE FROM (SYSTIMESTAMP - STARTED_AT)) * 60000    +
                                               EXTRACT(SECOND FROM (SYSTIMESTAMP - STARTED_AT)) * 1000)
                                          ELSE NULL END,
               ERROR_CODE          = p_error_code,
               ERROR_MESSAGE       = p_error_message,
               PAYLOAD_SNAPSHOT_REF = p_payload_ref,
               METADATA            = p_metadata
        WHERE  TRACKING_ID = v_tracking_id;

        -- Update pipeline-level counters
        IF p_new_status = 'COMPLETED' THEN
            UPDATE OIC_PIPELINE_RUNS
            SET    COMPLETED_STAGES = COMPLETED_STAGES + 1
            WHERE  CORRELATION_ID   = p_correlation_id;
        ELSIF p_new_status = 'FAILED' THEN
            -- [R-05 FIX] Only cascade OVERALL_STATUS to 'FAILED' when the failing
            -- stage is mandatory. Non-mandatory stage failures should NOT terminate
            -- the pipeline at the tracking layer. The canonical business pattern
            -- sends SKIPPED (not FAILED) for non-mandatory stage faults, but this
            -- guard defends against developers accidentally sending FAILED for
            -- a non-mandatory stage, which would incorrectly mark the entire run FAILED.
            UPDATE OIC_PIPELINE_RUNS
            SET    FAILED_STAGE_ID = p_stage_id,
                   OVERALL_STATUS  = CASE
                       WHEN (SELECT IS_MANDATORY
                             FROM   OIC_STAGE_TRACKING
                             WHERE  TRACKING_ID = v_tracking_id) = 'Y'
                       THEN 'FAILED'
                       ELSE OVERALL_STATUS   -- non-mandatory: preserve current status
                   END
            WHERE  CORRELATION_ID  = p_correlation_id;
        END IF;
    END IF;

    COMMIT;
    p_status    := 'SUCCESS';
    p_error_msg := NULL;

EXCEPTION WHEN OTHERS THEN
    ROLLBACK;
    p_status    := 'ERROR';
    p_error_msg := SQLERRM;
END SP_UPDATE_STAGE;


-- ------------------------------------------------------------
-- SP_COMPLETE_PIPELINE
-- Finalises pipeline run record
-- ------------------------------------------------------------
PROCEDURE SP_COMPLETE_PIPELINE (
    p_correlation_id   IN  VARCHAR2,
    p_overall_status   IN  VARCHAR2,
    p_completed_at     IN  VARCHAR2,
    p_status           OUT VARCHAR2,
    p_error_msg        OUT VARCHAR2
) AS
    v_completed_ts  TIMESTAMP;
BEGIN
    v_completed_ts := TO_TIMESTAMP(
                          p_completed_at,
                          'YYYY-MM-DD"T"HH24:MI:SS.FF3"Z"');

    -- [R-01 FIX] ASYNC_LAG_FLAG: set to 'Y' when elapsed time from INITIATED_AT
    -- to the queue-message completedAt exceeds 120 seconds. The business integration
    -- enqueues PIPELINE_COMPLETE at the true completion moment; v_completed_ts
    -- therefore reflects the real end time. A gap > 120s means the dispatcher
    -- was lagging — useful for dispatcher health dashboards and SLA monitoring.
    UPDATE OIC_PIPELINE_RUNS
    SET    OVERALL_STATUS  = p_overall_status,
           COMPLETED_AT    = v_completed_ts,
           ASYNC_LAG_FLAG  = CASE
               WHEN (v_completed_ts - INITIATED_AT) >
                    NUMTODSINTERVAL(120, 'SECOND')
               THEN 'Y'
               ELSE 'N'
           END
    WHERE  CORRELATION_ID = p_correlation_id
    AND    OVERALL_STATUS NOT IN ('FAILED'); -- don't override a FAILED status

    COMMIT;
    p_status    := 'SUCCESS';
    p_error_msg := NULL;

EXCEPTION WHEN OTHERS THEN
    ROLLBACK;
    p_status    := 'ERROR';
    p_error_msg := SQLERRM;
END SP_COMPLETE_PIPELINE;


-- ------------------------------------------------------------
-- SP_BATCH_WRITE_LOGS
-- Inserts array of log entries in one transaction
-- Key efficiency gain of async pattern
-- ------------------------------------------------------------
PROCEDURE SP_BATCH_WRITE_LOGS (
    p_log_entries      IN  CLOB,
    p_inserted_count   OUT NUMBER,
    p_status           OUT VARCHAR2,
    p_error_msg        OUT VARCHAR2
) AS
BEGIN
    INSERT INTO OIC_ACTIVITY_LOG (
        LOG_ID, CORRELATION_ID, TRACKING_ID, STAGE_ID,
        LOG_TIMESTAMP, SEVERITY, CATEGORY, ACTOR, ACTION,
        MESSAGE, DETAIL, STACK_TRACE, WRITTEN_AT, CREATED_AT
    )
    SELECT
        jt.log_id,
        jt.correlation_id,
        jt.tracking_id,
        jt.stage_id,
        TO_TIMESTAMP(jt.log_timestamp,
                     'YYYY-MM-DD"T"HH24:MI:SS.FF3"Z"'),
        jt.severity,
        jt.category,
        jt.actor,
        jt.action,
        jt.message,
        jt.detail,
        jt.stack_trace,
        SYSTIMESTAMP,
        SYSTIMESTAMP
    FROM JSON_TABLE(p_log_entries, '$[*]'
        COLUMNS (
            log_id         VARCHAR2(36)   PATH '$.logId',
            correlation_id VARCHAR2(36)   PATH '$.correlationId',
            tracking_id    VARCHAR2(36)   PATH '$.trackingId',
            stage_id       VARCHAR2(50)   PATH '$.stageId',
            log_timestamp  VARCHAR2(30)   PATH '$.timestamp',
            severity       VARCHAR2(10)   PATH '$.severity',
            category       VARCHAR2(20)   PATH '$.category',
            actor          VARCHAR2(200)  PATH '$.actor',
            action         VARCHAR2(200)  PATH '$.action',
            message        VARCHAR2(4000) PATH '$.message',
            detail         CLOB           FORMAT JSON PATH '$.detail',
            stack_trace    CLOB           PATH '$.stackTrace'
        )
    ) jt
    -- [R-03 FIX] WHERE NOT EXISTS makes SP_BATCH_WRITE_LOGS idempotent against
    -- OCI Queue at-least-once redelivery. Without this guard: if the dispatcher
    -- commits the INSERT but crashes before deleting the activity log messages
    -- from the queue, OCI Queue redelivers them after visibilityInSeconds (120s).
    -- The second INSERT would raise ORA-00001 on LOG_ID PK, failing the entire
    -- batch and routing all messages to DLQ — even though data was already saved.
    -- With this guard: redelivered messages are silently skipped (0 rows inserted),
    -- SQL%ROWCOUNT returns 0, the batch succeeds, messages are deleted from queue.
    WHERE NOT EXISTS (
        SELECT 1 FROM OIC_ACTIVITY_LOG
        WHERE  LOG_ID = jt.log_id
    );

    p_inserted_count := SQL%ROWCOUNT;
    COMMIT;
    p_status    := 'SUCCESS';
    p_error_msg := NULL;

EXCEPTION WHEN OTHERS THEN
    ROLLBACK;
    p_inserted_count := 0;
    p_status         := 'ERROR';
    p_error_msg      := SQLERRM;
END SP_BATCH_WRITE_LOGS;


-- ------------------------------------------------------------
-- SP_WRITE_DLQ
-- Records messages that failed all dispatcher retry attempts
-- ------------------------------------------------------------
PROCEDURE SP_WRITE_DLQ (
    p_queue_name       IN  VARCHAR2,
    p_message_id       IN  VARCHAR2,
    p_message_type     IN  VARCHAR2,
    p_correlation_id   IN  VARCHAR2,
    p_raw_payload      IN  CLOB,
    p_failure_reason   IN  VARCHAR2,
    p_status           OUT VARCHAR2,
    p_error_msg        OUT VARCHAR2
) AS
BEGIN
    MERGE INTO OIC_TRACKING_DLQ dlq
    USING (SELECT p_message_id AS msg_id FROM DUAL) src
    ON (dlq.ORIGINAL_MESSAGE_ID = src.msg_id)
    WHEN MATCHED THEN
        UPDATE SET
            ATTEMPT_COUNT   = dlq.ATTEMPT_COUNT + 1,
            LAST_FAILED_AT  = SYSTIMESTAMP,
            FAILURE_REASON  = p_failure_reason
    WHEN NOT MATCHED THEN
        INSERT (DLQ_ID, QUEUE_NAME, ORIGINAL_MESSAGE_ID, MESSAGE_TYPE,
                CORRELATION_ID, RAW_PAYLOAD, FAILURE_REASON,
                ATTEMPT_COUNT, FIRST_FAILED_AT, LAST_FAILED_AT)
        VALUES (SYS_GUID(), p_queue_name, p_message_id, p_message_type,
                p_correlation_id, p_raw_payload, p_failure_reason,
                1, SYSTIMESTAMP, SYSTIMESTAMP);

    COMMIT;
    p_status    := 'SUCCESS';
    p_error_msg := NULL;

EXCEPTION WHEN OTHERS THEN
    ROLLBACK;
    p_status    := 'ERROR';
    p_error_msg := SQLERRM;
END SP_WRITE_DLQ;

END OIC_TRACKER_PKG;

6.1 JS Helper Functions (embed as JS Action Library in OIC)
JavaScript
// ============================================================
// OIC GEN 3 TRACKING HELPER LIBRARY
// Embed as a JS Action at integration entry — assigns helpers
// to global scope variables for use throughout the flow
// ============================================================

// --- Core UUID generator (no external dependency) ---
function generateUUID() {
    return java.util.UUID.randomUUID().toString();
}

// --- Timestamp for queue messages ---
// [BUG-04 FIX] Explicitly set UTC timezone on SimpleDateFormat.
// Without setTimeZone(UTC), SimpleDateFormat uses the JVM default timezone.
// The OIC Gen 3 JVM may NOT be UTC depending on the region/instance configuration.
// A non-UTC JVM would produce timestamps like "2026-05-15T18:00:00.000Z" when the
// actual UTC time is 10:00 — the timestamps CLAIM to be UTC ('Z' suffix) but carry
// local time. This causes wrong DURATION_MS values and incorrect ordering in dashboards.
function nowISO() {
    var sdf = new java.text.SimpleDateFormat("yyyy-MM-dd'T'HH:mm:ss.SSS'Z'");
    sdf.setTimeZone(java.util.TimeZone.getTimeZone("UTC"));
    return sdf.format(new java.util.Date());
}

// --- Build PIPELINE_INIT queue message ---
// [BUG-05 FIX] corrId is now an EXPLICIT PARAMETER (was implicit outer-scope reference).
// In OIC's GraalVM/Nashorn JS engine, flow-scope variables from a previous JS action
// block are NOT automatically in scope inside helper function bodies. Referencing
// 'corrId' as a free variable inside a function defined in a different JS action
// results in ReferenceError or undefined. Pass corrId explicitly on every call.
// [ACC-05 FIX] stageManifest parameter REMOVED. The dispatcher reads the authoritative
// manifest from OIC_INTEGRATION_CONFIG via integrationId. A copy in the queue message
// is redundant and creates an inconsistency vector if the config is updated.
// [R-04 FIX] 'environment' added as explicit parameter. Source from OIC Lookup
// Table "OIC_INSTANCE_ENV" (key="THIS_INSTANCE") configured at deployment time.
// Valid values: "PROD" | "UAT" | "DEV". Defaults to "PROD" if not supplied.
function buildInitMessage(corrId, integrationId, businessKey, initiatedBy,
                          oicInstanceId, environment) {
    return JSON.stringify({
        messageId:      generateUUID(),
        messageType:    "PIPELINE_INIT",
        schemaVersion:  "1.0",
        correlationId:  corrId,
        integrationId:  integrationId,
        enqueuedAt:     nowISO(),
        payload: {
            businessKey:     businessKey,
            initiatedBy:     initiatedBy,
            oicInstanceId:   oicInstanceId,
            environment:     environment || "PROD"  // defaults to PROD if not provided
            // stageManifest intentionally absent: dispatcher reads from OIC_INTEGRATION_CONFIG
        }
    });
}

│  [GAP-06 FIX] The circuit breaker uses TWO stores in tandem:       │

Visibility Timeout Strategy:

Code
[GAP-07 FIX] OCI Queue visibilityInSeconds = 120 (was 30)
  The original 30s budget was insufficient under realistic load:
    - Object Storage PUT per FAILED stage:      200–500ms each
    - OIC Email action per FAILED stage:        300–800ms each
    - A batch with 5 failed stages adds:        2.5–6.5s of non-DB overhead
    - SP calls + network per message:           ~100–300ms each
    - Total for 10-msg batch with 5 failures:   up to 25–35s
  With 30s window, a heavy batch risks timeout before DELETE → redelivery.
  120s provides a safe ~4–5× margin over worst-case batch processing time.

[GAP-07 FIX] Tracking events batch size = 10 (was 20)
  Smaller batch + 120s window: worst-case processing guaranteed < 60s.
  Activity logs batch size remains DISPATCHER_BATCH_SIZE (default 20) since
  SP_BATCH_WRITE_LOGS has no Object Storage or Email overhead per message.

If dispatcher crashes mid-batch:
  → Messages become visible again after 120s
  → Next dispatcher run picks them up automatically
  → No message loss — at-least-once delivery guaranteed by OCI Queue
  → SP_INIT_PIPELINE / SP_UPDATE_STAGE are idempotent [BUG-06 FIX]
     (MERGE + WHERE NOT EXISTS patterns prevent duplicate records)

9. Data Retention & Housekeeping
SQL
-- ============================================================
-- RETENTION JOB — Scheduled nightly via DBMS_SCHEDULER
-- Respects per-integration RETENTION_DAYS config
-- ============================================================
BEGIN
    DBMS_SCHEDULER.CREATE_JOB (
        job_name        => 'OIC_TRACKER_RETENTION_JOB',
        job_type        => 'PLSQL_BLOCK',
        job_action      => '
            DECLARE
                v_cutoff TIMESTAMP;
            BEGIN
                FOR cfg IN (
                    SELECT INTEGRATION_ID, RETENTION_DAYS
                    FROM   OIC_INTEGRATION_CONFIG
                    WHERE  TRACKING_ENABLED = ''Y''
                ) LOOP
                    v_cutoff := SYSTIMESTAMP
                                - NUMTODSINTERVAL(cfg.RETENTION_DAYS, ''DAY'');

                    DELETE FROM OIC_ACTIVITY_LOG al
                    WHERE  al.CORRELATION_ID IN (
                               SELECT CORRELATION_ID
                               FROM   OIC_PIPELINE_RUNS
                               WHERE  INTEGRATION_ID = cfg.INTEGRATION_ID
                               AND    INITIATED_AT   < v_cutoff
                           );

                    DELETE FROM OIC_STAGE_TRACKING st
                    WHERE  st.CORRELATION_ID IN (
                               SELECT CORRELATION_ID
                               FROM   OIC_PIPELINE_RUNS
                               WHERE  INTEGRATION_ID = cfg.INTEGRATION_ID
                               AND    INITIATED_AT   < v_cutoff
                           );

                    -- [R-07] Non-partitioned path: these DELETEs are correct.
                    -- When OIC_PIPELINE_RUNS / OIC_STAGE_TRACKING / OIC_ACTIVITY_LOG
                    -- are partitioned by INITIATED_AT (see partitioning recommendation
                    -- in Section 4.1), replace all three DELETEs above with:
                    --   ALTER TABLE OIC_ACTIVITY_LOG   DROP PARTITION <partition_name>;
                    --   ALTER TABLE OIC_STAGE_TRACKING DROP PARTITION <partition_name>;
                    --   ALTER TABLE OIC_PIPELINE_RUNS  DROP PARTITION <partition_name>;
                    -- DROP PARTITION is O(1) metadata-only — orders of magnitude
                    -- faster than row-by-row DELETE for high-volume pipelines.
                    DELETE FROM OIC_PIPELINE_RUNS
                    WHERE  INTEGRATION_ID = cfg.INTEGRATION_ID
                    AND    INITIATED_AT   < v_cutoff;

                    COMMIT;
                END LOOP;
            END;',
        start_date      => SYSTIMESTAMP,
        repeat_interval => 'FREQ=DAILY;BYHOUR=2;BYMINUTE=0',
        enabled         => TRUE
    );
END;
